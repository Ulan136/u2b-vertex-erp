## 5. СЧЕТА — ШАБЛОНЫ И ПРИВЯЗКА

### 5.1 Четыре типа счетов (нумерованные)

| № | Название | Источники | Банковские счета |
|---|---------|-----------|-----------------|
| **1** | 📋 Поверка | САМИ + ВДК + ТЭЦ + Выездная | Каспи / БЦК / Наличка |
| **2** | 💰 Продажа | Продажа + Первичная поверка | Каспи / БЦК / Наличка |
| **3** | 📄 Прочие операции | Проекты + Услуги + Тендеры | **только БЦК** (авто) |
| **4** | 🏢 Филиал Астана | САМИ + Продажа | **Каспи Голд** |

### 5.2 Логика оплаты

- Тип счёта (1/2/3/4) — определяется **автоматически** по источнику данных
- Конкретный банковский счёт (Каспи/БЦК/Наличка) — **выбирает менеджер** при каждой операции
- Исключение: Счёт №3 «Прочие» → всегда только БЦК, без выбора
- Исключение: Счёт №4 «Филиал Астана» → всегда Каспи Голд

### 5.3 Конструктор шаблонов (Настройки → Привязка счетов)

Каждый шаблон содержит:
1. Номер, иконка, название, описание
2. Банковские счета (из пресета: Каспи/БЦК/Наличка/Каспи Голд/Другой) + реквизиты
3. Источники данных (вкл/выкл галочкой)
4. Тип, счёт по умолчанию, активность

Шаблоны синхронизируются с логикой всей системы автоматически.

### 5.4 Филиалы

- Каспи Голд (Астана) — **отдельный счёт**, не субсчёт
- Отдельный журнал операций
- Отдельная карточка в разделе Счета → Филиалы

## 6. БАЗА ДАННЫХ — АРХИВ И СРОКИ

### 6.1 Архивирование

- Записи старше **30 дней** автоматически уходят в архив
- Архив: только просмотр, поиск, фильтрация, экспорт
- Три архива: Сертификаты · Извещения · (будущее: Продажи)

### 6.2 Сроки счётчиков

Срок поверки: **5 лет** с даты поверки.

Статусы срока:
- 🔴 **Просрочено** — дата прошла
- 🟠 **Менее 3 месяцев** — критично
- 🔵 **3–6 месяцев** — скоро
- 🟢 **Норма** — всё хорошо

Данные из реестра Word — **все 19 записей с реальными данными**.

### 6.3 Авто-заказы

- Создаются автоматически при сроке < **90 дней**
- Поля: ФИО · Адрес · Тип · № счётчика · Срок · Причина · Мастер · Статус
- Статусы: Новый → В работе → Завершён
- Кнопка «Назначить» → выбор мастера + дата выезда

## 11. ИНТЕГРАЦИЯ С е-КТРМ (techreg.gov.kz)

### 11.1 Что такое е-КТРМ / ЭУПСИ

- **е-КТРМ** — государственная ИС технического регулирования РК
- **ЭУПСИ** — модуль «Электронный учёт данных о поверяемых СИ»
- Поверочные лаборатории **обязаны** вносить каждый сертификат
- Вход через **ЭЦП НУЦ РК** (NCALayer WebSocket)
- Подтверждение: 2 подписи — поверитель + руководитель

### 11.2 Кнопка 🤖 → КТРМ

Присутствует в **каждой таблице сертификатов** (САМИ, ВДК, ТЭЦ, Извещения).

Демо-режим (текущий):
- Анимированный лог терминала слева
- Симуляция браузера справа
- 8 шагов: NCALayer → авторизация → ЭУПСИ → поиск типа → заполнение полей → подпись → успех
- После демо: кнопка «✅ Подтвердить» → статус меняется на «Внесен в КТРМ»

### 11.3 Архитектура робота (Фаза 1 — Python RPA)

**Технология:** Python + Playwright + WebSocket к NCALayer

**Процесс:**
```
ERP (кнопка) → Python скрипт → Playwright открывает ЭУПСИ
→ Заполняет форму из данных ERP
→ Нажимает "Подписать"
→ NCALayer диалог: поверитель вводит пароль ОДИН РАЗ
→ Подпись №1 готова → следующий сертификат
→ После всех → Подпись №2 (руководитель)
→ Статус в ERP: ✅ Внесен в КТРМ
```

**По паролю:** рекомендован полуавтоматический режим — утром один раз вводится, сессия работает весь день.

**NCALayer API (JSON через WebSocket):**
```json
{
  "module": "kz.gov.pki.knca.commonUtils",
  "method": "signXml",
  "args": ["PKCS12", "SIGNATURE", "<xml_data>", "", ""]
}
```

expenses        -- расходы (cat_id, sub_cat)

## 18. ЛОГИКА СТАТУСОВ СЕРТИФИКАТА — ОЧЕРЕДЬ ДЛЯ РОБОТА

### 18.1 Концепция

Статусы поля **«Операция»** — это не просто метки. Это **очередь задач** для робота е-КТРМ.  
Робот периодически сканирует таблицу и обрабатывает записи по статусу автоматически.

```
Менеджер управляет первыми двумя шагами.
Робот управляет остальными.
```

### 18.2 Пять статусов и их логика

| # | Статус | Цвет | Кто меняет | Действие |
|---|--------|------|-----------|---------|
| 1 | 🔄 **В работе** | Серый | Система (авто при создании) | Данные заполняются менеджером |
| 2 | 📋 **Готова к КТРМ** | 🟠 Оранжевый | Менеджер вручную | Данные проверены, готово к отправке |
| 3 | 🤖 **Внести в КТРМ** | 🔵 Синий (пульс) | Менеджер вручную | **Команда роботу** — начать обработку |
| 4 | ⏳ **КТРМ 70%** | 🟢+🟠 полоска | **Робот** автоматически | Подпись №1 (поверитель) сделана |
| 5 | ✅ **Внесён в КТРМ** | 🟢 Зелёный | **Робот** автоматически | Подпись №2 (руководитель) сделана, готово |

### 18.3 Полный цикл обработки

```
┌─────────────────────────────────────────────────────────────┐
│                    МЕНЕДЖЕР                                  │
│                                                              │
│  1. Заполняет данные счётчика в форме                       │
│     → Статус: 🔄 В работе                                   │
│                                                              │
│  2. Проверяет данные, всё верно                             │
│     → Меняет на: 📋 Готова к КТРМ                          │
│                                                              │
│  3. Даёт команду роботу                                     │
│     → Меняет на: 🤖 Внести в КТРМ                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│                    РОБОТ е-КТРМ                              │
│                                                              │
│  4. Сканирует таблицу → видит "Внести в КТРМ"              │
│     → Открывает форму ЭУПСИ в браузере                      │
│     → Заполняет все поля из данных ERP                      │
│     → Вызывает NCALayer → Подпись №1 (поверитель)          │
│     → Меняет статус на: ⏳ КТРМ 70%                        │
│                                                              │
│  5. Сканирует таблицу → видит "КТРМ 70%"                   │
│     → Открывает форму подтверждения в ЭУПСИ                 │
│     → Вызывает NCALayer → Подпись №2 (руководитель)        │
│     → Меняет статус на: ✅ Внесён в КТРМ                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 18.4 Batch-режим (пакетная обработка)

Робот не ждёт команды на каждую запись — он обрабатывает **все** записи со статусом "Внести в КТРМ" подряд:

```python
# Псевдокод логики робота
while True:
    records = db.query("SELECT * FROM certs WHERE oper = 'Внести в КТРМ'")
    for rec in records:
        fill_eupsi_form(rec)          # заполнить форму
        sign_with_ncalayer(cert_key)  # подпись №1
        db.update(rec.id, oper='КТРМ 70%')
    
    records_70 = db.query("SELECT * FROM certs WHERE oper = 'КТРМ 70%'")
    for rec in records_70:
        confirm_eupsi(rec)            # подтвердить
        sign_with_ncalayer(boss_key)  # подпись №2
        db.update(rec.id, oper='Внесён в КТРМ')
    
    sleep(30)  # проверять каждые 30 секунд
```

### 18.5 Преимущества этой архитектуры

- **Менеджер не ждёт робота** — поставил статус и работает дальше
- **Робот работает в фоне** — обрабатывает очередь пока менеджер занят другим
- **Отказоустойчивость** — если робот упал, записи остаются в статусе "Внести в КТРМ" и будут обработаны при следующем запуске
- **Прозрачность** — по таблице сразу видно что сделано, что в процессе, что ждёт
- **Пакетная обработка** — утром поставил 19 записей в очередь → ушёл на обед → вернулся → все внесены
- **Две подписи разделены** — поверитель и руководитель могут подписывать в разное время

### 18.6 Интеграция с ERP (прототип → продакшн)

| Этап | Хранение статуса | Обновление |
|------|-----------------|-----------|
| **Прототип (сейчас)** | JavaScript массив в памяти | Вручную через клик в попапе |
| **Sprint 2 (Next.js)** | PostgreSQL поле `oper` в таблице `certificates` | API endpoint `PATCH /api/certs/:id` |
| **Sprint 4 (Робот)** | PostgreSQL | Python скрипт обновляет через API после каждого действия |
